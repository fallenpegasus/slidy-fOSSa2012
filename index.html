We know what makes good software

  • Decoupled Design
  • Understood Interfaces
  • Iterative Development
  • Code Review
  • Solid Testing
  • Happy Programmers

Closed Source process

  • "Business Needs" drive developement
  • Feature release on meaningless deadlines
  • Architecture breaks down
  • Developers don't have time or passion
  • Code review breaks down
  • Difficult process
  • Yet too easy to commit

Technical Debt

  • Never time to do things right.
  • Rarely time to do things over.
  • Very bad software is kept alive because it makes money.
Open Source at different scales:

One programmer

  • Does it because he "has an itch"
  • Probably going to make it work.
  • Won't make it too complex to understand.
  • Exists as long as it's useful.

One maintainer plus contributors.

  • He will code review everything.
  • He's not going to break it.
  • If it remains useful, someone will maintain it
  • Contributors mean useful features
  • Contributors means "Linus's Law"
Linus's Law:
With enough eyeballs,
all bugs are shallow.

Small Team

  • Will work on code that makes them happy.
  • Can't force work they don't want.
  • Encourages decoupled design.

Large team, on large projects.

  • Can Open Source possibly work?
  • Lots of failures
  • Lots of very painful successes
  • Lots of lessons learned.
Big Disclaimer

GCC & Gnu BinUtils

  • Tightly coupled team
  • Tightly coupled design
  • Resistant to outside contributors.

Conway's Law

  • The organization of the software and the organization of the software team will be congruent.
  • "If you have four groups working on a compiler, you'll get a 4-pass compiler".
  • "Loosely coupled teams will result in loosely coupled software".

Linux

  • Really Big "One developer with contributors"
  • We got lucky.
  • Cannot clone Linuses.
  • We discovered Linus's Law
  • We discovered DVCS

Java & JCP

  • Developed "corporate style"
  • Unhappy developers
  • Dominated by one company
  • Closed test, QA, cert

JCP & ASF Java

  • "Deep rich APIs"
  • APIs coupled to implementation
  • Huge piles of "donated" code
  • Uninvolved "maintainers"
  • Discovered that big companies can collaborate

MySQL

  • Dominated by one company
  • Tightly coupled design
  • Features driven by Marketing Goals
  • Feature Release instead of Time Release
  • Hard to build
  • Very resistant to outside contributions
  • Code review fell apart

We Learned Lessons

  • Easy to get involved
  • Easy to contribute
  • Easy development tooling
  • Loose coupling
  • Code Review is critical
  • Collaboration tooling
  • Face to Face Meetings

OpenStack

  • Not dominated by one company
  • Business goals cannot override the development process
  • Code review is central
  • APIs are HTTP REST WSGI
  • Very loosely coupled
  • Time Based Release
  • Human can't commit to trunk

Open Participation

  • Open Design Summits, in Person
  • Open Meetings, in IRC

Gated Trunk

  • Ensures Code Quality
  • Protects developers
    • Always start from working code
  • Protects tree
    • Bad code never merges
  • Egalitarian
    • Same for everyone
    • Transparent
    • Automated

Open Source works when

  • Code Review
  • Easy participation
  • Avoid corporate anti-patterns
